# Radio-86RK programmer's reference

A single reference for writing or reversing Intel 8080 code on the
Radio-86RK. The RK86's monitor ROM is not swappable ‚Äî it defines
the hardware contract (variable slots, keyboard scan, video init,
tape format), so this document treats the monitor as an integral
part of the machine rather than a separate layer.

Sources:

- Monitor ROM: [begoon/rk86-monitor](https://github.com/begoon/rk86-monitor/blob/main/monitor.asm)
- Interactive viewer: <https://rk86.ru/monitor>
- Assembler: [asm8](https://github.com/begoon/asm8) ‚Äî browser + CLI
  i8080 assembler. Same engine the emulator bundles as the asm8
  playground (toolbar "–ê—Å—Å–µ–º–±–ª–µ—Ä") and the terminal tool uses via
  `asm8080` on npm. Produces ready-to-load `.rk` files.

## Memory map

| Range       | Contents                                        |
|-------------|-------------------------------------------------|
| 0000h‚Äì75FFh | Free user RAM (30 KB on a 32 KB machine)        |
| 7600h‚Äì765Fh | Monitor variables ([see below](#monitor-variables-7600h765fh)) |
| 7660h‚Äì76CFh | Monitor stack (grows down from 76CFh)           |
| 76D0h‚Äì7FF4h | Video memory ‚Äî 30 rows √ó 78 cols = 2340 B       |
| 8000h‚Äì8003h | PPI D20 (keyboard) ‚Äî –ö580–í–í55                   |
| A000h‚ÄìA003h | PPI D14 (tape / ROM-disk) ‚Äî –ö580–í–í55            |
| C000h‚ÄìC001h | CRT controller ‚Äî –ö580–í–ì75                       |
| E004h‚ÄìE008h | DMA ‚Äî –ö580–í–¢57                                  |
| F000h       | ROM extension slot (`U` command jumps here)     |
| F800h‚ÄìFFFFh | Monitor ROM (2 KB)                              |

`memory_top = 75FFh` is the documented upper bound of free user
memory; programs typically start with `lxi sp, 75FFh` (or any lower
address). The monitor itself runs with `SP = 76CFh`.

## Peripherals

### Keyboard (–ö580–í–í55 "D20" at 8000h)

| Addr  | Role                                                  |
|-------|-------------------------------------------------------|
| 8000h | Port A ‚Äî row scan mask (output, active low)           |
| 8001h | Port B ‚Äî column data (input, active low)              |
| 8002h | Port C ‚Äî modifier keys + tape I/O                     |
| 8003h | Control word (monitor programs `8Bh` at cold start)   |

Port C bits (`8002h`, active low):

- bit 5 ‚Äî SHIFT (–°–°)
- bit 6 ‚Äî CTRL (–£–°)
- bit 7 ‚Äî –Ý–£–°/–õ–ê–¢
- bits 0‚Äì3 ‚Äî tape in/out and speaker

Matrix scan idiom:

```asm
    mvi  a, 0FDh        ; select row 1 (0 = active)
    sta  8000h
    lda  8001h          ; read columns
    cma                 ; 1 = pressed
```

Prefer the monitor's `getc` / `kbhit` / `scan_kbd` over raw matrix
reads ‚Äî they handle autorepeat, –Ý–£–°/–õ–ê–¢, and debouncing.

### CRT controller (–ö580–í–ì75 at C000h)

| Addr  | Role                                   |
|-------|----------------------------------------|
| C000h | Parameter / data register              |
| C001h | Command register (W) / status (R)      |

After a reset the screen is blank until the CRT is reprogrammed.
Call `F82Dh` (monitor `video`) to reinitialize it; cold start does
this automatically. The sequence emitted is:

| Write              | Decodes as                                               |
|--------------------|----------------------------------------------------------|
| `C001h ‚Üê 00h`      | **Reset** command ‚Äî 4 parameter bytes follow on C000h    |
| `C000h ‚Üê 4Dh`      | SCN1: 78 characters per row (H=77, no row spacing)       |
| `C000h ‚Üê 1Dh`      | SCN2: 30 rows per frame (V=29, VR=0)                     |
| `C000h ‚Üê 99h`      | SCN3: 10 scan lines per char row, underline at line 9    |
| `C000h ‚Üê 93h`      | SCN4: line-counter mode + blinking full-block cursor     |
| `C001h ‚Üê 27h`      | **Start Display** command ‚Äî DMA burst/spacing parameters |

Seeing the raw pattern `00 4D 1D 99 93 ‚Ä¶ 27` in a binary is almost
always a manual video reinit. Cursor blink is hardware-driven at
**500 ms** period.

### Tape / ROM-disk PPI (–ö580–í–í55 "D14" at A000h)

| Addr  | Role                          |
|-------|-------------------------------|
| A000h | Port A ‚Äî ROM-disk data        |
| A001h | Port B ‚Äî ROM-disk addr low    |
| A002h | Port C ‚Äî ROM-disk addr high   |
| A003h | Control word                  |

Most programs go through the monitor's `inpb` / `outb` / `inpblock` /
`outblock` entries rather than touching this PPI directly.

### Sound

The RK86 has no dedicated sound chip ‚Äî the speaker is wired to
Port C bit 3 of the tape PPI (A002h). Tone is produced by toggling
that bit at the desired audio frequency. The emulator detects the
toggles on the 8080's interrupt-enable line (`EI`/`DI` pattern that
some programs use as a period marker) and synthesizes a square-wave
tone whose frequency is `1.78 MHz / (ticks_between_toggles √ó 2)`.

`putc 07h` ("beep") is a short tone produced this way; games drive
their own sound by timed `EI`/`DI` or bit-3 toggles in a delay loop.

## Keyboard matrix

Full 8√ó8 matrix (write `~(1<<row)` to `8000h`, read `8001h`, invert).
Cells name the **RK86 keycap**, not a PC key:

| Row | Bit 0     | Bit 1   | Bit 2   | Bit 3 | Bit 4 | Bit 5 | Bit 6 | Bit 7 |
|-----|-----------|---------|---------|-------|-------|-------|-------|-------|
| 0   | ‚Üñ (home)  | –°–¢–Ý     | –ê–Ý2     | –§1    | –§2    | –§3    | –§4    | ‚Äî     |
| 1   | TAB       | `` ` `` | –ü–°/CR   | –ó–ë/BS | ‚Üê     | ‚Üë     | ‚Üí     | ‚Üì     |
| 2   | 0         | 1       | 2       | 3     | 4     | 5     | 6     | 7     |
| 3   | 8         | 9       | `*`/`:` | `;`   | `,`   | `-`   | `.`   | `/`   |
| 4   | `–Æ`/`@`   | A/–ê     | B/–ë     | C/–¶   | D/–î   | E/–ï   | F/–§   | G/–ì   |
| 5   | H/–•       | I/–ò     | J/–ô     | K/–ö   | L/–õ   | M/–ú   | N/–ù   | O/–û   |
| 6   | P/–ü       | Q/–Ø     | R/–Ý     | S/–°   | T/–¢   | U/–£   | V/–ñ   | W/–í   |
| 7   | X/–¨       | Y/–´     | Z/–ó     | `[`   | `\`   | `]`   | `'`   | SPACE |

Row 0 is the non-alphanumeric "command" row. The real RK86 has only
**four function keys** (–§1..–§4); the rest of row 0 hosts ‚Üñ
(home-cursor), –°–¢–Ý (clear-screen) and –ê–Ý2 (starts the monitor's
`ESC Y y x` cursor-positioning sequence). Letter cells show the two
glyphs sharing that keycap, `LAT/–Ý–£–°`. Row 3 bit 2 and row 4 bit 0
are the two non-letter keycaps that swap glyphs with –Ý–£–°/–õ–ê–¢:
`*`‚Üî`:` and `–Æ`‚Üî`@`.

Row 1 has the control-code keys; reading it with `FDh` on port A
gives back these codes:

| Bit | Key          | Code |
|-----|--------------|------|
| 0   | TAB          | 09h  |
| 1   | LF           | 0Ah  |
| 2   | CR (Enter)   | 0Dh  |
| 3   | DEL          | 7Fh  |
| 4   | Left arrow   | 08h  |
| 5   | Up arrow     | 19h  |
| 6   | Right arrow  | 18h  |
| 7   | Down arrow   | 1Ah  |

The emulator remaps modern PC keys onto this matrix ‚Äî e.g. PC
`F5`..`F9` cover row 0's –ê–Ý2 / –§-keys / home / –°–¢–Ý, PC `F10`
toggles –Ý–£–°/–õ–ê–¢. Those are emulator conveniences, not RK86 hardware.

## Monitor ROM

The monitor ROM lives at F800h‚ÄìFFFFh. Its jump table at F800h is
the stable ABI ‚Äî call those slot addresses, not the `entry_*`
labels, since internal entry points can move between ROM revisions.

### Stack

Monitor sets `SP = 76CFh` (just below video memory at 76D0h). Stack
grows downward into the 7660h‚Äì76CEh area. User programs should use
the same stack or set SP to a safe address below 7600h.

|-------|-------------------|------|--------------------------------------------|
| 7600h | `cursor_addr`     | 2    | Video-RAM address of the cursor cell       |
| 7602h | `cursor_xy`       | 2    | `(Y,X)` ‚Äî H=row, L=col                     |
| 7604h | `ap2_seq`         | 1    | Byte index in an –ê–Ý2 escape sequence       |

| 7614h | `pc_storage`      | 2    | Saved PC (G-command / breakpoint)          |
| 7616h | `hl_storage`      | 2    | Saved HL                                   |
| 7618h | `bc_storage`      | 2    | Saved BC                                   |
| 761Ah | `de_storage`      | 2    | Saved DE                                   |
| 761Ch | `sp_storage`      | 2    | Saved SP                                   |
| 761Eh | `af_storage`      | 2    | Saved A+PSW                                |
| 7623h | `stop_addr`       | 2    | Breakpoint address                         |
| 7625h | `stop_addr_value` | 1    | Original byte replaced by the breakpoint   |
| 7626h | `jump_to_go`      | 3    | `C3 xx xx` trampoline rewritten by `G`     |
| 7627h | `cmd_param_1`     | 2    | 1st parameter of the active command        |
| 7629h | `cmd_param_2`     | 2    | 2nd parameter                              |
| 762Bh | `cmd_param_3`     | 2    | 3rd parameter                              |
| 762Dh | `param_2_present` | 1    | FFh if the 2nd parameter was supplied      |
| 762Eh | `tape_polarity`   | 1    | Detected polarity of the tape input        |
| 762Fh | `tape_read_const` | 1    | Default `2Ah` (‚âà591 ¬µs / bit)              |
| 7630h | `tape_write_const`| 1    | Default `1Dh` (‚âà406 ¬µs / bit)              |
| 7631h | `memory_limit`    | 2    | Top-of-RAM reported by `getlim`            |
| 7633h | `kbd_buf`         | 32   | Keyboard ring buffer                       |
| 76CFh | `stack_bottom`    | ‚Äî    | Monitor SP                                 |

`cursor_addr` at 7600h is handy for direct-to-video tricks: `lhld
7600h`, then write bytes through `(HL)`. `putc` updates it for you.

`jump_to_go` at 7626h is how user code is launched. The `G` command
writes the target into 7627h/7628h and does a plain `JMP 7626h`.
The slot doubles as the return-to-breakpoint vector: the `RST 6`
handler pushes flags/PC then returns through this trampoline so you
can resume with another `G`.

`ap2_seq` at 7604h tracks monitor escape-sequence state. Only
`ESC Y y x` is implemented: `1Bh` arms the sequence, the next byte
must be `Y`, and the two bytes after that set `row+20h`, `col+20h`.
Any other second byte aborts the sequence and is printed literally.

Non-jump-table entry points worth knowing:

| Addr  | Name         | Notes                                             |
|-------|--------------|---------------------------------------------------|
| F836h | entry_start  | Cold start (banner + prompt)                      |
| F86Ch | prompt_loop  | `JMP F86Ch` = "exit back to monitor prompt"       |

ROM routines are not register-preserving ‚Äî save anything you need
across a call. `putc` in particular touches PSW, B, D, H internally.

Typical end-of-program pattern:

```asm
    lxi h, msg
    call 0F818h      ; puts
    jmp 0F86Ch       ; back to monitor prompt

msg: db "hello, world", 0
```

### Monitor commands

The prompt accepts a single letter followed by up to three hex
arguments separated by spaces or commas, terminated by Enter (or `.`
to abort back to the prompt). `?` is echoed on a parse error.

| Cmd | Args                   | Action                                                         |
|-----|------------------------|----------------------------------------------------------------|
| `D` | `start,end`            | Dump memory as hex, 16 bytes per line                          |
| `L` | `start,end`            | List memory as text (printable chars; others shown as `.`)     |
| `F` | `start,end,byte`       | Fill `start..end` inclusive with `byte`                        |
| `S` | `start,end,byte`       | Search `start..end` for `byte`, report addresses               |
| `T` | `src_start,src_end,dst`| Transfer (copy) `src_start..src_end` to `dst`                  |
| `C` | `a_start,a_end,b_start`| Compare two blocks, report differences                         |
| `M` | `addr`                 | Modify memory ‚Äî shows current byte, type new one or Enter/`.`  |
| `G` | `addr[,breakpoint]`    | Go ‚Äî jump to `addr`, optionally set a breakpoint first         |
| `I` | `[offset]`             | Input block from tape; optional tape-constant `offset`         |
| `O` | `start,end[,const]`    | Output block `start..end` to tape; optional write-timing const |
| `R` | `start,end`            | Read block from ROM-disk (PPI D14 at A000h)                    |
| `X` | ‚Äî                      | Print saved registers (after G or a breakpoint)                |
| `U` | ‚Äî                      | Jump to `F000h` (ROM-extension slot)                           |

`G`'s breakpoint is implemented by patching `RST 6` (`F7h`) at the
target address; the original byte is saved in `stop_addr_value`
(7625h). When the breakpoint fires, the monitor saves all registers
into the 7614h‚Äì761Eh storage, restores the byte, and returns to the
prompt. `X` prints that saved state.

## Video memory

```text
rows 0..29  √ó  78 columns per row  =  2340 bytes at 76D0h

video address(y, x) = 76D0h + y*78 + x
```

Each byte is a **character code**, not a pixel pattern ‚Äî the –í–ì75
fetches the glyph from the character-generator ROM. The default
text window used by the monitor covers **columns 8..71, rows 3..27**
(64 √ó 25). Its top-left is at `77C2h = 76D0h + 3*78 + 8`.

## Character set

One byte per cell, **KOI-7 N2** ‚Äî the 7-bit Soviet standard that
KOI-8 is the 8-bit superset of. Russian uppercase lives at 60h‚Äì7Fh
(KOI-7 N2's Cyrillic half); the same letters appear at E0h‚ÄìFFh in
KOI-8. Converting between them is a bit-7 flip: `koi8 = koi7 | 80h`.
On the RK86 those KOI-8 codes instead select the inverse-video copy
of the glyph, so strings are always written in the 60h‚Äì7Fh range.

| Range     | Contents                                                 |
|-----------|----------------------------------------------------------|
| 00h‚Äì1Fh   | Block/line graphics and arrows (many cells render blank) |
| 20h‚Äì5Fh   | Standard ASCII                                           |
| 60h‚Äì7Fh   | Cyrillic uppercase (Latin ‚Üí Russian mapping below)       |
| 80h‚ÄìFFh   | Inverse video of 00h‚Äì7Fh                                 |

## Control codes and ESC-Y

| Code | Effect (via `putc`)                             |
|------|-------------------------------------------------|
| 07h  | Beep (tone duration from B/C at routine entry)  |
| 08h  | Cursor left                                     |
| 0Ah  | Line feed                                       |
| 0Ch  | Home cursor                                     |
| 0Dh  | Carriage return                                 |
| 18h  | Cursor right                                    |
| 19h  | Cursor up                                       |
| 1Ah  | Cursor down                                     |
| 1Bh  | Start of escape sequence (see below)            |
| 1Fh  | Clear screen (and home the cursor)              |

Only one escape sequence is honored: `ESC Y y x`, where `y` and `x`
are the target row and column each **biased by +20h**. So to move
to row 5, column 10:

```asm
    mvi  c, 1Bh
    call 0F809h
    mvi  c, 'Y'
    call 0F809h
    mvi  c, 5 + 20h
    call 0F809h
    mvi  c, 10 + 20h
    call 0F809h
```

Newline in strings is CR+LF: the two bytes `0Dh, 0Ah`.

## Keyboard I/O

`getc` at `F803h` blocks until a key is pressed and returns one of:

| Code       | Key                                                   |
|------------|-------------------------------------------------------|
| 00h        | Ctrl-@ (NUL) ‚Äî used as `F4` in some programs          |
| 01h, 02h   | Ctrl-A / Ctrl-B ‚Äî extra function-key slots            |
| 03h        | Ctrl-C ‚Äî monitor programs treat it as "reset"         |
| 08h        | Cursor left / BS                                      |
| 09h        | Tab                                                   |
| 0Ah / 0Dh  | LF / CR (Enter)                                       |
| 18h        | Cursor right                                          |
| 19h        | Cursor up                                             |
| 1Ah        | Cursor down                                           |
| 20h‚Äì7Fh    | Printable character (Latin or Cyrillic per –Ý–£–°/–õ–ê–¢)   |

`kbhit` at `F812h` is the non-blocking variant ‚Äî returns 00h if
nothing is pressed, otherwise the key code. `scan_kbd` at `F81Bh`
returns FFh for "no key", FEh for the –Ý–£–°/–õ–ê–¢ toggle alone, or the
raw key code.

## Tape

Programs almost always go through the monitor:

```asm
    lxi  h, start
    lxi  d, end
    call 0F827h        ; outblock - write HL..DE, BC=checksum out
    ...
    lxi  h, 0          ; 0 = use address from the block header
    call 0F824h        ; inpblock - read, CY set on checksum error
```

Low-level constants worth knowing:

| Value | Meaning                                     |
|-------|---------------------------------------------|
| E6h   | Sync byte ‚Äî also a literal marker in files  |
| 2Ah   | Default read timing constant (‚âà591 ¬µs/bit)  |
| 1Dh   | Default write timing constant (‚âà406 ¬µs/bit) |

Bit framing ‚Äî every bit is written as **two edges** on the tape
output line. The monitor emits the two edges roughly `1Dh` ticks
apart for `0` bits and with different spacing for `1` bits; the
receiver times edge-to-edge intervals and reconstructs the bit.
Bytes are 8 bits MSB-first. A block on tape has this structure:

```text
[256 √ó 00h lead-in]  E6h  start_hi start_lo  end_hi end_lo
                     [payload bytes]
                     E6h  sum_hi sum_lo
```

The 256-byte lead-in lets the receiver lock onto the bit timing
before the first `E6h` sync appears.

## File formats

Address fields are 2-byte **big-endian**; `end_addr` is the address
of the **last** byte (inclusive, not one-past-the-end). `payload`
is `end_addr ‚àí start_addr + 1` bytes of raw i8080 code/data ‚Äî
identical to what you'd get from `.bin`. `checksum` is the monitor's
`chksum` routine (`F82Ah`) run over `start_addr..end_addr`, stored
big-endian.

Checksum algorithm ‚Äî **not** a plain 16-bit sum. For a buffer of
`N` bytes `v[0..N-1]`:

```text
sum = 0
for i in 0..N-2:                       # every byte EXCEPT the last
    sum = (sum + v[i] + (v[i] << 8)) & 0xFFFF
# final byte folds into the low half only:
sum = (sum & 0xFF00) | ((sum + v[N-1]) & 0x00FF)
```

Equivalently: maintain two bytes `(hi, lo)`. For every byte except
the last, `lo += v[i]` and `hi += v[i] + carry_from_lo`. For the
last byte, only `lo` is updated, discarding any carry. This quirk
(last byte treated specially, checksum NOT a commutative sum) is
what makes the RK86 checksum distinguishable from a naive
accumulator when reversing.

| Extension     | Layout                                                                 |
|---------------|------------------------------------------------------------------------|
| `.bin`        | Raw payload only, no header or trailer.                                |
| `.rk`, `.rkr` | `start_hi start_lo end_hi end_lo` ‚Äñ `payload` ‚Äñ `E6 sum_hi sum_lo`     |
| `.pki`, `.gam`| `E6 start_hi start_lo end_hi end_lo` ‚Äñ `payload` ‚Äñ `E6 sum_hi sum_lo`  |

So `.pki` / `.gam` and `.rk` / `.rkr` differ only in the leading
sync byte: `.pki` / `.gam` repeat the `E6h` sync byte before the
address header, `.rk` / `.rkr` omit it. The trailing
`E6 sum_hi sum_lo` is the same in both.

Examples ‚Äî a 3-byte payload `21 00 00` (`lxi h, 0`) at address
3000h, checksum `0021h`:

```text
.bin   21 00 00
.rk    30 00 30 02 | 21 00 00 | E6 00 21
.pki   E6 30 00 30 02 | 21 00 00 | E6 00 21
```

This is exactly the `.rk` tape-file format the playground emits.
Programs are loaded at the address encoded in the file header; the
asm8 `org` directive is what sets that address.

## Programming idioms

### Program skeleton

```asm
    org  3000h             ; any address in 0000h..75FFh
    lxi  sp, 75FFh         ; stack at the top of user RAM

    lxi  h, hello
    call 0F818h            ; puts
    jmp  0F86Ch            ; back to the monitor prompt

hello: db 'hi, world', 0Dh, 0Ah, 0
```

### Clear screen + print at (row, col)

```asm
    mvi  c, 1Fh            ; clear screen + home
    call 0F809h
    mvi  c, 1Bh            ; ESC
    call 0F809h
    mvi  c, 'Y'
    call 0F809h
    mvi  c, 3 + 20h        ; row 3
    call 0F809h
    mvi  c, 8 + 20h        ; col 8
    call 0F809h
    lxi  h, msg
    call 0F818h
```

A reusable `set_cursor` that patches an ESC-Y template in place
(preserves nothing by default) is how `docs/examples/dumped.asm`
does it.

### Read one key

```asm
wait:   call 0F803h         ; getc, blocks
        cpi  0Dh            ; Enter
        jz   done
        cpi  1Bh            ; ESC
        jz   abort
        ...
```

### Direct video write at current cursor

```asm
    lhld 7600h              ; HL = current cursor address
    mvi  m, 17h             ; 17h = full block
```

Direct writes bypass `putc`, so the monitor's own cursor tracking
stays in sync only if you follow up with a `putc` or an `ESC Y`
sequence.

### Exit back to the monitor

```asm
    jmp  0F86Ch             ; warm restart (keeps screen state)
    ; or
    jmp  0F800h             ; cold restart (reinitializes everything)
```

Pressing Ctrl-C during `getc` in a well-behaved program should route
to `0F800h` (the hard reset).

## Reversing cheat sheet

Patterns you will see over and over in RK86 binaries:

| Sequence                          | Meaning                        |
|-----------------------------------|--------------------------------|
| `31 FF 75`                        | `lxi sp, 75FFh` ‚Äî stack init   |
| `CD 18 F8`                        | `call puts`                    |
| `CD 09 F8`                        | `call putc`                    |
| `CD 03 F8`                        | `call getc`                    |
| `CD 15 F8`                        | `call hexb` ‚Äî print A as 2 hex |
| `C3 6C F8`                        | `jmp` to monitor prompt        |
| `0E 1F CD 09 F8`                  | clear screen                   |
| `0E 1B CD 09 F8 0E 59 CD 09 F8 ‚Ä¶` | ESC-Y cursor positioning       |
| `2A 00 76`                        | `lhld 7600h` ‚Äî cursor address  |
| `3A 06 76`                        | `lda 7606h` ‚Äî –Ý–£–°/–õ–ê–¢ flag     |
| Any `CPI 03h / JZ F800h`          | "Ctrl-C returns to monitor"    |

Control-code checks on `getc` results (`CPI 08h / 18h / 19h / 1Ah`)
almost always indicate cursor-key handling.

Self-modifying `MVI A, nn` followed by a `STA ??+1` elsewhere is a
common idiom for storing a tiny amount of state inline ‚Äî the dump
editor uses this to remember "mid-byte edit in progress" without
spending a RAM byte.

### i8080 opcode field layout (for manual disassembly)

Register index in `mov`, `mov r,r'`, `mvi`, arithmetic-with-r:

| r | 0 | 1 | 2 | 3 | 4 | 5 | 6     | 7 |
|---|---|---|---|---|---|---|-------|---|
|   | B | C | D | E | H | L | M=(HL)| A |

Register-**pair** index in `lxi rp`, `push`/`pop`, `dad`, `inx`/`dcx`,
`ldax`/`stax`:

| rp | 00 | 01 | 10 | 11 |
|----|----|----|----|----|
|    | BC | DE | HL | SP |

`push`/`pop` substitute `PSW` for the `11` slot (so `11` = SP for
`LXI`/`DAD`/`INX`/`DCX` but PSW for `PUSH`/`POP`).

### Undocumented/alias opcodes

The i8080 leaves several unused opcodes which behave as NOP but are
distinct in a disassembler's eyes ‚Äî spotting them often means "this
byte is actually data, not code":

| Bytes                                          | Nominal behavior     |
|------------------------------------------------|----------------------|
| `08h`, `10h`, `18h`, `20h`, `28h`, `30h`, `38h`| NOP (alias)          |
| `CBh`                                          | JMP (alias of `C3h`) |
| `D9h`                                          | RET (alias of `C9h`) |
| `DDh`, `EDh`, `FDh`                            | CALL (alias of `CDh`)|

### Half-carry on AND (`ANA`) is unusual

Unlike ADD/SUB where half-carry is computed from bit 3 of
`(A + op)`, `ANA r` / `ANI n` sets HF from bit 3 of **`A | r`**
(logical OR of the operand and the accumulator). This is a real
i8080 quirk documented in the Intel 8080 Programmer's Manual ‚Äî
code that branches on HF after an AND is relying on that formula,
not on the result.

### Russian strings typed as Latin

Old sources often store Cyrillic text by typing it on a Latin
keyboard and letting the 60h‚Äì7Fh range render the right glyphs. The
bytes look like gibberish in a hex dump but decode cleanly through
the Cyrillic row table above:

| Source bytes (as Latin) | Render on-screen       |
|-------------------------|------------------------|
| `pozdrawlq` s pobedoj`  | `–ü–û–ó–î–Ý–ê–í–õ–Ø–Æ –° –ü–û–ë–ï–î–û–ô` |
| `prigotowxtesx`         | `–ü–Ý–ò–ì–û–¢–û–í–¨–¢–ï–°–¨`        |

When you see a string that is almost-ASCII but peppered with
backticks, `q`, `x`, `w`, `{`, `|`, `}`, `~`, assume it's Russian
and translate via the 60h‚Äì7Fh mapping.
